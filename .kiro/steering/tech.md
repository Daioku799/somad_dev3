# Technology Stack

## Architecture

**モジュール化されたハイブリッド・モデリング・アーキテクチャ**
- データのパース、コンポーネント配置計算、幾何計算（判定カーネル）、モデル構築（格子生成・統合）、視覚化（検証プロット）を独立したモジュールに分離。
- 外部ファイル（JSON/GDSII）による宣言的なモデル定義と、手続き的な格子マッピングを統合。

## Core Technologies

- **Language**: Julia
- **Runtime**: Julia 1.10+
- **Paradigm**: 構造体ベースの不変データ管理とマルチディスパッチ。

## Key Libraries

- **JSON.jl**: 設定ファイルのパースと統合管理。
- **SimpleGDS.jl**: GDSIIファイルのバイナリパース（低レベル処理）。
- **PolygonOps.jl**: GDSII由来의 ポリゴンに対する内外判定。
- **LinearAlgebra.jl**: 幾何サンプリング、距離計算、外挿座標算出。
- **Plots.jl**: モデル検証および結果描画。

## Development Standards

### Type Safety
- **不変構造体 (Immutable struct)**: 設定データ、物理定数、および幾何形状（GdsPolygon等）は `struct` で定義し、不慮の書き換えを防止。
- **厳格な型指定**: 浮動小数点数（Float64）と整数（Int）を明示的に区別し、計算精度を維持。

### Code Quality
- **境界検証**: 外部からの入力値（λなど）はロード時にバリデーションを行い、不正な値での解析開始を阻止。
- **多層的視覚検証**: 構築されたモデルの正当性を保証するため、主要な断面（XY, YZ）の自動プロット生成を必須とし、物理構造（TSVの貫通等）を目視で確認可能にする。
- **GDS重ね書き検証**: 格子マッピングの正確性を検証するため、IDマップのヒートマップ上にソースGDSの境界線をオーバーレイ描画し、座標の不整合を視覚的に検出する。
- **動的非一様格子生成**: 静的なハードコードを排除し、層境界マーカーと物理オフセット（pg_dpth等）から数学的に格子面（Faces）を導出する。
- **ソルバー互換性維持**: 解析エンジンが要求するガイドセル（+2サイズ）および外挿座標（nk+3）の仕様を、データ構造レベルで厳格に維持する。
- **マスター座標同期**: TSVやバンプといった垂直スタックを形成する要素に対し、共通のマスター座標リストを共有し、層を跨いだ幾何学的同期を保証する。
- **物理的整合性バリデーション**: 3Dグリッド充填前に、チップ境界逸脱やコンポーネント間干渉を数学的に検証し、物理的に不可能な配置を早期に検出する。
- **セルベース判定カーネル**: 「単一点」ではなく「セルの境界」を受け取り、占有率 50% を閾値とする数学カーネルを判定の基本単位とする。
- **BBox早期棄却**: サンプリング等の重い幾何計算の前に、境界ボックス（BBox）による重なりチェックを必須とし、パフォーマンスを最適化する。
- **幾何学的整合性の自動修正**: GDSIIデータ等の幾何入力に対し、重複頂点の除去や閉路補完をロード時に自動実行し、判定の安定性を確保。
- **再現性**: ランダムシード（random_seed）の管理により、配置の再現性を確保。

### Testing
- **Julia Test Framework**: `using Test` を用いたユニットテストおよび統合テスト。
- **誤差許容比較**: 物理計算結果の比較には `isapprox` (≈) を使用。

## Development Environment

### Common Commands
```bash
# Test: julia test/run_all.jl
# Run: julia src/main.jl  (将来的なメインエントリ)
```

## Key Technical Decisions

- **デフォルト値の排除**: 解析実行時はJSONでの明示的指定を必須とし、自動補完による暗黙的な解析を避ける（エラー停止を優先）。
- **試験用初期値ユーティリティ**: 開発・テスト用にのみ `H2-main-original` 準拠のデータを生成する専用パスを用意。
